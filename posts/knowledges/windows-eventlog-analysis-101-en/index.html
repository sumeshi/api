How Do You Investigate Windows Event Logs?

\n

Vague notes on Windows event log investigation. I'll add more as I remember.

\n

\n

Introduction

\n

I wanted something like a document for doing Windows event log investigations, so here it is.
\nMainly from an incident response perspective.

\n

The purpose of event log investigation is not to comprehensively scoop up every suspicious event ID, but to grasp what happened in the target environment and connect it to initial response and further investigation.
\nGiven that premise, keep the following in mind at minimum.

\n
    \n
  • Event logs only tell you about what's still around.
      \n
    • Quite a lot of events aren't recorded by default. Depends on the audit policy.
      • \n
    • Old event logs get overwritten. Depends on the rotation settings.
      • \n
    • Logs can also be deleted by the attacker. There are even cases where deletion is automated in ransomware incidents, so sometimes you can't tell a thing from the surviving logs alone.
      • \n
    • The possibility of log tampering can't be ruled out either. EVTX has a checksum mechanism for integrity checking, but it's not a cryptographic tamper-protection mechanism. It costs more than simple deletion, so you don't see it much.
      • \n
    • From the above, it's important for an analyst to not treat surviving event logs as standalone facts, but to build and verify hypotheses about the attacker's actions while cross-referencing other artifacts.
      • \n
    \n
    • \n
  • Preserve logs first thing
      \n
    • Even while you're reading this article, new events are being recorded and old ones overwritten.
      • \n
    • The Security log, which is especially important evidence, gets overwritten fast. You're lucky if a few days' worth survives, and sometimes only a few hours' worth is left. (Like when it's buried under logon failures.)
      • \n
    • If you've got time to hesitate, preserve first.
      • \n
    \n
    • \n
  • How to cut noise out of a massive pile of event logs? Hugely important.
      \n
    • Be prepared to look at a few GB per client machine, and tens to hundreds of GB per server.
      • \n
    • Understanding how the device is normally used and what events it records is the first step in cutting noise.
      • \n
    • \"Just look at everything, you say?\" I will punch you.
      • \n
    \n
    • \n

Purpose

\n

You can't investigate logs without knowing what you're trying to confirm.
\nBreak down from the big goal and clarify the investigation's purpose.

\n

Whether it's 5W1H or the Cyber Kill Chain or whatever, organizing around some framework makes it harder to miss things.

\n
    \n
  • Did an intrusion occur? (Detection)
    • \n
  • Where did they get in? (Initial access)
    • \n
  • How did it spread? (Lateral movement / Persistence)
    • \n
  • What did they do? (Activity)
    • \n
  • What was affected? (Impact scope)
    • \n
  • Why was it possible? (Cause)
    • \n
\n

Stuff like that.

\n

Well, the reason you want to investigate is usually \"whether there was an intrusion\", and if there was, \"whether there was data leakage\" or something along those lines.
\nSo what do you need to look at to investigate that? What's recorded in logs is ultimately concrete events, so you have to make a high-confidence guess from circumstantial evidence about what happened from the chain of those events.

\n

For example, if you want to know \"whether there was data leakage\", the event log is never going to literally say \"data was leaked\". What you should look at is the traces of actions that could lead to leakage \u2014 suspicious logons, file access, external communication, removable media connections, access to admin shares, execution of suspicious tools, and so on.

\n

Cutting Scope

\n

Log investigation has no end (seriously).
\nResources vanish in proportion to how much you throw at it, so cutting scope up front is important.

\n

If the results show the scope was too narrow, you can think about it then.

\n
    \n
  • Which hosts' logs to investigate? (Servers? Clients?)
    • \n
  • Which time period? (The intrusion period?)
    • \n
  • Which logs? (Security? System? Application? PowerShell? TaskScheduler?)
    • \n
  • Which users' logs? (Admins? Regular users? Service accounts?)
    • \n
  • Which event IDs? (Logon events? Process creation events? Network connection events?)
    • \n
\n

Knack for Investigation

\n

For starters, go through the major event IDs once.
\nAfter that, build hypotheses based on them and think about what to investigate.

\n

Interpreting logs does rely on experience to some extent... But basically it's best to refer to reliable official docs like MS Learn.

\n

Also, if you have Sysmon you can look at finer events too,,, but in most environments it isn't installed. Don't get your hopes up.

\n

The Security Log Encyclopedia is also super helpful. There are a lot of undocumented events, so if you're curious about an event ID, look it up there.

\n

If you still can't find it, sometimes you can find it on MyEventlog. Surprisingly, EventID is an insane spec that isn't a unique identifier, but this site properly shows search results per Provider/Channel.

\n

Even more niche ones are sometimes written up in the exabeam Documentation. If it's still not there, you have no choice but to Google like crazy.

\n

If you know the tools or techniques used, it's good to look for traces while referring to things like JPCERT/CC's Tool Analysis Result Sheet.

\n

You can also look at Yamato Security's Guide to Windows Event Log Settings for DFIR and Threat Hunting. It references Sigma and discusses what the default event log settings look like. If possible, look at the original Sigma rule definitions, but there are so many you couldn't possibly memorize them.

\n

Log Analysis and Hypothesis Building

\n

Once you've looked at the logs to some extent, build hypotheses about what might have happened, what attack techniques are conceivable, and what damage is conceivable.
\nTo verify those hypotheses, dig deeper into what other events to look for and what artifacts to cross-reference.

\n

If you don't know how to investigate, you can ask generative AI. But don't paste logs as-is. Since they may contain sensitive info, always mask the input and follow your organization's rules.
\nAlso, if you ask a slightly deeper question, keep in the back of your mind that the accuracy isn't that great (or rather, it leans too hard toward generalities).

\n

Reporting

\n

Report what might have happened as a result of the investigation and what to do next.
\nOr, if that's not the analyst's job, it's good to clarify where another team's territory begins. In incident response, pushing nasty work onto each other happens often enough, so negotiating it in advance makes things smoother.

\n

Also, for the materials you hand to the report recipients, you have to hammer them into your brain enough that you can explain everything no matter what you're asked.\nFor everything else, just keep materials on hand so you can answer if asked.

\n

If there's a second opinion involved or the customer has security expertise, you might get asked \"Did you check \u25ef\u25ef?\". Resources are limited so you don't need to prepare for everything, but be able to explain why you didn't check that item. \"I excluded it from the investigation for this and that reason. It's out of contract scope.\" and so on.

\n

Also, it's fine to narrow down what you report. Rather, there is absolutely no need to report every hypothesis conceivable from the investigation results.
\nTo avoid needlessly worrying or confusing the other party with baseless hypotheses, it's enough to focus on the highest-confidence hypothesis and the investigation you did to verify it.

Appendix. Major Event IDs

\n

Rather than looking at logs comprehensively, it's better to look at them from the perspective of finding a starting point.
\nOnce you find a suspicious event as a starting point, diligently tracing the timeline before and after it should turn up something useful even for unknown events.

\n

Logon / Authentication

\n

Look at who, when, from where, logged onto which machine (lateral movement).
\nStart investigating from the device where the problem occurred, or the jump server that's the entry point from the internet.

\n

When looking, be conscious of whether the recorded event is from the logon source or the logon destination.

\n

  • Security.evtx

    • 4625: Logon failure

    • 4634: User logoff completed

\n

The LogonType of event ID 4624 is extremely important. Basically you'll look mainly at 3 (Network) and 10 (RemoteInteractive), with 9 (NewCredentials) and 12 (CachedRemoteInteractive) as support.

\n

For logon failures in event ID 4625, the logon error code tells you why it failed. Like the username doesn't exist in the first place, or the username is right but the password is wrong.. This tells you how much information the attacker had at the time they did it. For example, if they logged on successfully on the first try, they're dumping passwords somewhere or reusing the same one.

Logon TypeDescription
0Logon type used only by the SYSTEM account.
1No information. On Reddit, the rumor is it's a leftover from the NT 3.x era.
2Interactive. Logon where a user interactively uses the machine. Console logon, RUNAS, remote shell, KVM, operation via Lights-Out cards, IIS Basic Auth (before 6.0), etc.
3Network. Logon to access a target over the network. Shared access via net use, MMC snap-in to a remote computer, PowerShell WinRM, PsExec, Remote Registry, Remote Desktop Gateway, vulnerability scanners, IIS Integrated Windows Auth, SQL Windows Auth, etc. LogonUser does not cache credentials for this logon type. Reusable credentials, as a rule, do not remain in the destination LSA session, but watch out for exceptions like when Kerberos delegation is enabled. PsExec, when explicit credentials are specified, can create both Network and Interactive logon sessions.
4Batch. Logon for running a process on behalf of a user without their direct operation. Scheduled tasks, etc. Also used for high-performance server purposes like mail/web servers that process many plaintext auth attempts at once. LogonUser does not cache credentials for this logon type. However, in scheduled tasks, the password may be stored on disk as an LSA Secret.
5Service. Logon by a service. The target account needs the \"Log on as a service\" right. Credentials for running the service can remain in the LSA session as reusable credentials. Also, the password may be stored on disk as an LSA Secret.
6Proxy (proxy logon). Officially described as a proxy-type logon. On Reddit, the rumor is it's for internal/dev builds only.
7Unlock. Workstation unlock. A logon type for recording the unlock of a user interactively using the machine, like via GINA DLL.
8NetworkCleartext (network plaintext auth). Holds the name and password in the auth package, and the server can connect to other network servers while impersonating the client. IIS Basic Auth (6.0+), PowerShell WinRM with CredSSP, etc. Reusable credentials remain on the destination side, so the credential-theft risk is high.
9NewCredentials. Clones the current token and specifies different credentials for outbound network connections. Locally keeps the original identity, and only uses the specified credentials for network connections. RUNAS /NETWORK, etc. Reusable credentials can remain in the LSA session, so be careful.
10RemoteInteractive (remote interactive). Remote and interactive terminal services session. Remote Desktop, etc. Not only successful RDP logons, but 4625 logon failures can also be recorded as RemoteInteractive. Reusable credentials remain in the destination LSA session, so privileged account RDP to a compromised machine is dangerous.
11CachedInteractive (cached interactive). Interactive logon using cached credentials without accessing the network. Not necessarily a logon that authenticated by querying a domain controller.
12CachedRemoteInteractive (cached remote interactive). Same as RemoteInteractive. For internal audit purposes.
13CachedUnlock (cached unlock). Workstation unlock using cached credentials.

Log Rotation

\n

You might not use this much in investigation, but knowing it makes it easier to judge why there's a gap in the logs when you see one.

\n

For event logs, you can specify per Channel \"maximum size\" and \"what to do when full\". The latter has the following three options.

\n
    \n
  1. Overwrite events as needed (overwrite old events as needed)
    2. \n
  2. Archive the log when full, do not overwrite events (archive the full file and record to a new log file)
    3. \n
  3. Do not overwrite events / Clear log manually (don't record new events when full)
    4. \n
\n

1 reuses old log space in a FIFO manner to write new events. So old RecordID/RecordNumber numbers appear to be missing. Don't jump to \"attacker deletion\" just because there's a gap \u2014 check the rotation settings and think about why it's gone.

\n

2 archives (auto-backs-up) the full log while creating a new file and keeps recording. As an investigator this is the happiest option, but naturally it eats up capacity fast. If you want to design log management properly, rather than hoarding it on local disk, you should design it to forward to an event log collection server or SIEM for storage. Also in this case, 1105 is recorded in the Security log.

\n

3 stops recording new events once full. This setting is genuinely a pain, please stop using it. What possible joy does it bring. Also in this case, 1104 is recorded in the Security log.

\n
    \n
  • Security.evtx
      \n
    • 1104: The security log is full
      • \n
    • 1105: The event log was automatically backed up
      • \n

Choosing the Right Tool

\n

For Non-Analysts

\n

\n

Event Viewer

\n

If you're a field engineer or device administrator and you're told to look at the logs first and check whether there might be an intrusion, start here.

\n

Use the Windows built-in Event Viewer.
\nYou can search and filter in the GUI, so it's plenty usable for an initial check.

\n

However, do NOT hand a CSV export to CSIRT or an analyst. Seriously, stop.
\nFrom the investigator's side, they have to start by reshaping that garbage CSV into something usable, which is extra work. Also a lot of information is lost. If you're handing it to the investigator, preserve it in .evtx format, not .csv.

\n

\n

Example of a baffling CSV file exported from Event Viewer

\n

If you're collecting data with fast forensics in mind, you can use something like CyLR. Click the exe a few times and you can grab the major data in one go.
\nHowever, if you hand it to an overseas security vendor, they might go \"what's that?\", so ask which tool to collect with. Maybe KAPE, Velociraptor or something.

\n

If you're manually doing one or two files, Event Viewer's \"Save All Events As\" is fine, or you can export with wevtutil epl.

\n

The target files are under C:\\Windows\\System32\\winevt\\Logs\\, but copying the file can fail due to locks, so if possible it's safer to get them via the proper export procedure.

\n

Example:

\n
> wevtutil epl Security C:\\Temp\\Security.evtx\n> wevtutil epl System C:\\Temp\\System.evtx\n> wevtutil epl Application C:\\Temp\\Application.evtx\n
\n

Analysts

\n

If you're CSIRT or a forensic investigator, pick whichever method you like.
\nHowever, it's good to agree within the team on what to use.

\n

Even if you're a contrarian nerd who loves using a different tool from everyone else, don't forget that you have to be able to explain that the tool can guarantee evidentiary integrity.

\n

You need to be able to explain how the tool you used parses what, what format it outputs, and how it maps to the original log.

\n

Parsing with EvtxEcmd

\n

CSV is the most handable format. You can grep it too.\nConvert with file spec -f or folder spec -d. With a folder spec, the .evtx files underneath are merged into a single .csv.

\n
> EvtxECmd.exe -f Security.evtx --csv . --csvf Security.csv\n
\n

\n

Filtering with Timeline Explorer

\n

\n

You can filter with conditions like Event Id = 4624, click columns to sort, group, and do almost anything you can think of.
\nYou can't automate it, but for a \"let's just look at a bunch of stuff\" moment, this alone is fine.

\n

Filtering with Quilter-CSV

\n

Quilter-CSV is a tool I made myself and I'm pretty happy with it.

\n

You can chain multiple processes like this. For simple processing, this alone is enough.

\n
> qsv load Security.csv - isin EventId 4624 - select TimeCreated,EventId,PayloadData1 - sort TimeCreated - head 10 - showtable\n
\n

\n

Alternatively, you can process along a pre-defined yaml file.
\nThis is nicer when you want to apply it to multiple files.

\n
> qsv quilt test.yaml --var input=Security.csv\n
\n
title: test\n\nstages:\n  load_stage:\n    type: process\n    steps:\n      load:\n        path: ${input}\n\n  filtering:\n    type: process\n    source: load_stage\n    steps:\n      isin:\n        colname: EventId\n        values:\n          - "4624"\n      select:\n        colnames:\n          - TimeCreated\n          - EventId\n          - PayloadData1\n      sort:\n        colnames:\n          - TimeCreated\n      head:\n        number: 10\n\n  output_stage:\n    type: process\n    source: filtering\n    steps:\n      showtable:\n
